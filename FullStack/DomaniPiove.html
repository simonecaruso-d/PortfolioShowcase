01 - Problem

Tomorrow's forecast is one provider's guess

Weather apps show you a single number with false precision. No context, no accuracy history, no way to know if you should trust the "70% chance of rain" for your Saturday plans

Before DomaniPiove

  • |Every weather app shows one provider's prediction as if it were fact
  • |No visibility into how accurate that provider has been at that lead time
  • |No cross-provider comparison to understand forecast uncertainty
  • |The question "can I plan outdoor activities?" has no structured answer
  • |Accuracy degrades at longer lead times but apps never tell you by how much

After DomaniPiove

  • →Four independent providers aggregated and compared side-by-side
  • →Every provider scored by real error against historical observations
  • →Accuracy tracked across seven lead-time buckets: you see how confidence decays
  • →"Can I go out?" score translates weather data into an actionable decision
  • →AI concierge contextualizes the forecast for your specific plans

Strategic framing: the insight is that a single forecast without provenance is unactionable. Once you know which provider tends to be most accurate at a 3-day lead time in your city, that forecast earns weight. DomaniPiove builds the infrastructure to make that provenance explicit and persistent: then surfaces it in a UI designed for decisions

02 - Architecture

Six layers, one living database

Unlike a stateless pipeline, DomaniPiove has persistent state: forecasts are stored the moment they're fetched, actuals arrive later, and the accuracy engine joins them continuously. Every layer is independently automatable and testable

Why a database: DomaniPiove's data is temporal: forecasts are issued days before actuals arrive. Accuracy can only be computed after the fact, which requires durable storage. Supabase provides managed PostgreSQL with composite indexes on city-datetime pairs, giving sub-second joins on the forecast-actual merge that drives every accuracy metric

04 - Accuracy Engine

Scoring providers against reality

Forecasts are promises. Actuals are truth. The accuracy engine runs every evening, joining past predictions against observed weather and computing two rigorous error metrics across seven lead-time buckets for every provider

Metric 01

Mean Absolute Error

Average of absolute differences between forecast and actual. Expressed in the original unit (°C, mm, km/h). Interpretable and outlier-resistant.

MAE = mean( abs(forecast − actual) )
Metric 02

Mean Absolute Percentage Error

Percentage-based error normalized by the actual value. Enables cross-variable comparison and provider benchmarking on the same scale.

MAPE = mean( abs(forecast − actual) / actual )

Lead-time accuracy buckets: metrics computed independently for each span

07 - Operations

Three workflows, zero manual intervention

The product is fully automated: data refreshes every morning, accuracy is recomputed every evening, and the deployment stays alive without human input. All schedules are aligned to Europe/Rome business time

Workflow 01
UpdateSupabase
Daily · 06:00 UTC (8 AM Rome)

What it does

Calls all four weather APIs, normalizes each response to the common schema, and writes fresh forecasts to Supabase. Runs every morning so users always see today's predictions when they open the app

FetchProviders.py
workflow_dispatch

Secrets used

Supabase URL · Supabase API key · Visual Crossing API key · Pirate Weather API key, all injected from GitHub repository secrets at runtime

Workflow 02
RunForecastAccuracy
Daily · 18:00 & 19:00 UTC → gate at 20:00 Rome

What it does

Joins the last 14 days of stored forecasts against observed actuals, computes MAE and MAPE per provider per lead-time bucket, and writes results back to Supabase. Runs in the evening after actuals are available for the day

CalculateForecastAccuracy.py
14-day trailing window

Scheduling nuance

Triggered at two UTC times to account for daylight saving transitions in Rome. A timezone check inside the workflow gates execution to exactly 20:00 Rome time, preventing double-runs and ensuring consistent data across seasons

Workflow 03
KeepStreamlitWarm
Periodic ping

What it does

Sends a lightweight request to the Streamlit Cloud deployment to prevent the free-tier idle spin-down. Ensures users land on an immediately responsive app rather than waiting for a cold start

Streamlit Cloud
Keep-warm pattern

Why it matters

Streamlit Cloud's free tier suspends apps after inactivity. A cold start can take 30–60 seconds and destroys the first impression. This workflow eliminates that latency entirely for regular users

Timezone-aware scheduling: a naive UTC cron for an Italian-facing product would drift by an hour every DST transition. Using a Rome-timezone gate inside the workflow means the data lifecycle stays aligned to local business time regardless of UTC offset changes across the year, a small but operationally important detail

08 - Design

The choices that shaped the system

Each architectural decision reflects a deliberate tradeoff

Storage
Supabase vs. in-memory DataFrames
Persistence layer choice

Chosen: Supabase PostgreSQL

Forecasts and actuals arrive at different times and must be joined days later. Stateless in-memory processing can't bridge that temporal gap

Trade-off accepted

More infrastructure complexity, a DB dependency, and credential management. Mitigated by Supabase managed hosting and concurrent session-level caching that hides DB latency from users

Aggregation
Four providers vs one authoritative source
Forecast sourcing strategy

Chosen: Four independent providers

Single-model forecasts have structural blind spots. Aggregating independent models with different underlying physics provides measurably lower expected error and visible uncertainty signals when providers disagree

Trade-off accepted

4× the API surface area, 4× the failure modes, and schema unification complexity. Justified because provider disagreement is itself a signal: high variance among providers tells the user that confidence is genuinely low

AI layer
OpenRouter vs direct LLM provider
AI concierge integration

Chosen: OpenRouter abstraction

Model-agnostic routing with a fallback chain: if the primary model hits rate limits or an outage, the next model in the chain takes over without any code change

Trade-off accepted

One additional dependency and slightly higher per-call latency vs a direct API. Accepted because model switching (cost optimization, capability upgrades) becomes a configuration change rather than a code change

Accuracy
SCD snapshot forecasts vs. overwrite-latest
Historical data strategy

Chosen: SCD-like versioned snapshots

Each daily fetch preserves the forecast as it existed on that date, enabling temporal analysis: you can see how a provider's 7-day prediction evolved as the target date approached

Trade-off accepted

Higher storage volume and more complex join logic vs a simpler overwrite model. The extra data is what makes the accuracy engine meaningful: overwriting would erase the evidence

09 - Stack

Technology choices

Every tool chosen for a reason

Language
Python 3.12
Unified language across ingestion, analytics, accuracy engine, and UI. No context-switching, no cross-language serialization
UI Framework
Streamlit 1.40
Data products belong in Python. Custom CSS, fragments, and cache_data close the gap to a bespoke frontend at a fraction of the complexity
Database
Supabase · PostgreSQL
Managed PostgreSQL with REST API, composite indexes, and JSONB. Zero-infrastructure persistence with strong typing and sub-second queries
Data Processing
Pandas · NumPy
Forecast-actual joins, unpivot, groupby, MAE/MAPE aggregations: all vectorized without reinventing primitives
Visualization
Plotly · Altair · Folium
Plotly for interactive charts, Altair for declarative statistical views, Folium for geographic maps. Each picked for the specific chart type it handles best
HTTP Client
httpx · requests
httpx for async-capable concurrent provider fetching with timeout control. Exponential backoff implemented at the fetch layer
AI Gateway
OpenRouter
Model-agnostic routing with fallback chain. Swap or upgrade models without touching integration code
CI/CD · Ops
GitHub Actions
Three workflows cover data refresh, accuracy computation, and deployment keep-warm. Business-timezone-aligned scheduling with DST handling
Deploy
Streamlit Cloud
Zero-config deploy from repo. Secrets management, SSL, and scaling handled by the platform
Config
python-dotenv
Clean separation of secrets from code. Same pattern works locally and in GitHub via repository secrets
Dev Environment
Dev Containers
Reproducible local environment. Any contributor gets the same Python version, dependencies, and tooling with one command
Styling
Custom CSS
Glassmorphism aesthetic with fixed topbar, gradient backgrounds, and smooth micro-interactions. Makes the product feel intentional, not scaffolded
