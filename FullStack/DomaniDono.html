01 - Problem

Donating blindly is the default

Italian nonprofit data exists publicly but is fragmented, raw, and impossible for a regular person to act on. I built DomaniDono to close that gap and allow people to make better-informed decisions

Before DomaniDono

  • |Donors face thousands of NGOs with no comparative framework
  • |Open data exists but requires technical skill to parse and normalize
  • |Impact is not measured: only legal registration and revenue figures
  • |Peer benchmarking is impossible without building an engine from scratch
  • |No one translates the numbers into an actionable recommendation

After DomaniDono

  • →Multi-dimension ranking surfaces best candidates across 7 axes
  • →Pipeline normalizes open data into decision-ready DataFrames automatically
  • →Impact quadrant classifies every organization by reach and beneficiary scale
  • →Year-over-year deltas and peer benchmarks exposed in one click
  • →AI concierge turns structured data into a personalized recommendation

Strategic framing: the insight driving the whole design is that the problem isn't a lack of data, but a lack of translation. Every product decision flows from this: pipeline normalizes → engine computes → UI visualizes → AI recommends. Each layer serves the one above it

02 - Architecture

Five layers, one product

Designed as a linear pipeline where each layer produces a clean output for the next. No database: pure in-memory DataFrames with caching. Modular enough to swap any layer independently.

st.cache_data throughout
Streamlit fragments for isolated re-render
GitHub Actions CI/CD

Why Streamlit, not React: pipeline and UI share the same runtime, the same DataFrames, the same language: no API layer, no serialization, no split deploy. A custom web stack would triple the complexity with zero user-visible benefit at this scale. Hard module boundaries (app / db / configuration) keep each layer independently testable and swappable without that tradeoff changing

Resilience

Maintainability & Scalability

Every layer is independently swappable. Add a data source, swap the AI model, or extend the UI without touching the rest of the stack

Hard module boundaries: db / app / configuration
Stateless engine: composable and testable in isolation
Model-agnostic AI layer: swap providers without code changes
Pipeline designed to ingest additional data sources

AI design decisions: OpenRouter over a direct provider gives a model-agnostic fallback chain: if the primary model hits a rate limit or outage, the fallback kicks in without touching the integration code. Ranking table injected directly as prompt context rather than RAG: the dataset is bounded, structured, and deterministic. Direct injection is faster, cheaper, and fully auditable: when output is wrong, you can trace exactly what data the model saw

07 - Stack

Technology choices

Every tool chosen for a reason

Language
Python 3.11+
Unified language for data pipeline, analytics, and UI
UI Framework
Streamlit
Data products belong in Python. Fragments + cache_data + custom CSS get 90% of React with 10% of the complexity
Data Processing
Pandas
Vectorized ops, groupby, pct_change, transform: all the analytics primitives without writing them from scratch
AI Gateway
OpenRouter
Model-agnostic routing with fallback chain. Swap models without touching integration code
Data Source
Open Cooperazione
Italian government open data for nonprofits: authoritative, structured, and free to use commercially
CI/CD
GitHub Actions
Automated pipeline for tests and deploy. Foundation for scheduled data refresh jobs
Deploy
Streamlit Cloud
Zero-config deploy from repo. Handles secrets, scaling, and SSL
Styling
Custom CSS
Streamlit's default UI communicates "prototype". Custom CSS communicates "product". The difference matters to users
