HubSpot Generic Retrieval

A reusable paginated wrapper for extracting any HubSpot object into clean Pandas DataFrames with one function call

Built as a companion tool to the Deals pipeline: where the deals script is purpose-built for a complex schema, this is the general-purpose layer for any HubSpot endpoint that follows the standard paginated response pattern

HubSpot API Paginated REST Pandas Process Automation
01 - Problem

Every HubSpot object needed its own export

Owners, products, meetings, line items required manual exports. The paginated response pattern is identical across all of them. The solution was one wrapper, not five scripts.

Pattern recognition: HubSpot's v3 API returns a consistent envelope across all object types: results array, paging.next.after cursor, and a top-level length field. Once you write pagination correctly once, every endpoint becomes a one-liner. The generic wrapper captures this invariant

Without wrapper
  • |Manual exports from HubSpot website
  • |Separate pagination logic duplicated per script
With wrapper
  • →One HubSpotRequest function handles all paginated endpoints
  • →New object = one new URL + property string
  • →Pagination fixed once, works everywhere
  • →All outputs follow the same DataFrame → Excel pattern
02 - The Wrapper

One function, any HubSpot object

HubSpotRequest takes a URL, headers, and a payload and returns a fully paginated DataFrame. The pagination loop handles cursor extraction and concatenation automatically

01
Initial Request
First page fetch
  • GET to provided URL
  • Parse results array
  • Extract paging cursor
  • Read total length signal
02
Pagination Loop
While length > 1
  • Append cursor to URL
  • Fetch next page
  • Concat to running DataFrame
  • Update cursor if more pages
03
Output
Flat DataFrame
  • json_normalize flattens nested JSON
  • All pages concatenated
  • ignore_index resets row numbers
  • Ready for export or downstream use

Termination condition: the loop uses len(dataJson) > 1 as the "more pages" signal: when HubSpot has no further results, the response envelope shrinks to just the results key with no paging object. This avoids a KeyError on paging.next.after when exhausted.

04 - Core Pattern

The full wrapper in 18 lines

The entire reusable logic: pagination, concatenation, and normalization

HubSpotGenericDataRetrieval.py Python 
def HubSpotRequest(url, headers, payload):
    response = requests.request('GET', url, headers=headers, data=payload)
    dataJson = response.json()
    results  = dataJson['results']
    after    = dataJson['paging']['next']['after']
    lenght   = len(dataJson)
    df       = pd.DataFrame.from_dict(pd.json_normalize(results), orient='columns')

    while lenght > 1:
        responseTwo = requests.request('GET', f'{url}&after={after}', headers=headers, data=payload)
        dataJsonTwo = responseTwo.json()
        resultsTwo  = dataJsonTwo['results']
        lenght      = len(dataJsonTwo)
        dfTwo       = pd.DataFrame.from_dict(pd.json_normalize(resultsTwo), orient='columns')
        df          = pd.concat([df, dfTwo], ignore_index=True)
        if lenght > 1: after = dataJsonTwo['paging']['next']['after']

    return df

# One call per entity type
OwnersDf    = HubSpotRequest('https://api.hubapi.com/owners/v2/owners', headers, payload)
LineItemsDf = HubSpotRequest('https://api.hubapi.com/crm/v4/objects/line_items/?properties=name&...', headers, payload)
