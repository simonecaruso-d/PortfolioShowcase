HubSpot Deals Pipeline

A production-grade data extraction engine that retrieves, enriches, and exports the full deal universe from HubSpot CRM

Built to replace error-prone manual exports: 100+ properties per deal, paginated batch APIs for associated companies and contacts, and deterministic Excel output that matches an existing reporting schema exactly

HubSpot API v3 Process automation Schema Enforcement
01 - Problem

Manual CRM exports break at scale

HubSpot's UI export caps fields, loses time-in-stage data, and produces inconsistent column schemas. Sales reporting across three pipelines needed a reliable, automatable extraction layer

Before the pipeline

  • |Manual UI exports capped at a subset of available fields
  • |Time-in-stage metrics unavailable outside the API
  • |Stage, pipeline, and team IDs left as raw numeric keys
  • |Associated companies and contacts required separate lookups
  • |Timezone inconsistencies between sources and reporting
  • |No schema enforcement: column order and types varied by export

After the pipeline

  • →All 100+ properties fetched via API with dynamic property resolution
  • →Cumulative + latest time-in-stage for all 22 stages, ms → HH:mm:ss
  • →All IDs resolved to human-readable labels at runtime
  • →Company and contact names batch-resolved (100 IDs per request)
  • →All datetimes normalized to Europe/Rome, floored to minute
  • →Output schema locked to reference file column order and types

Design constraint: the output Excel had to match an existing reporting schema exactly. This drove the schema-lock pattern: load a reference file on startup, use it to derive required properties, enforce column order, and coerce types at the end

03 - Pipeline

API → Normalized DataFrame

Each step is a pure transformation on the DataFrame. No state shared between steps except the running DataFrame

01
Property Discovery
Dynamic field resolution
  • Query HubSpot property definitions API
  • Filter to schema-required columns only
  • Build name → label mapping at runtime
  • Inject time-in-stage computed keys
02
Deal Ingestion
Paginated cursor fetch
  • 100 deals per request (API max)
  • Cursor-based pagination loop
  • Properties + associations in one call
  • Flatten JSON to DataFrame via json_normalize
03
Association Resolution
Batch company + contact lookup
  • Extract all unique company/contact IDs
  • Batch POST: 100 IDs per request
  • Primary + all names resolved separately
  • Deduplication on ID level
04
Enrichment
ID → label + time conversion
  • Stages, pipelines, teams: ID → label via API maps
  • Owners resolved via owner map
  • ms → HH:mm:ss for all 44 time-in-stage columns
  • Record source + forecast → human labels
05
Normalization
Schema lock + type coercion
  • Column order locked to reference file
  • Bool / numeric / datetime types enforced per reference schema
  • Datetimes: UTC → Europe/Rome → floor minute
  • Export to dated Excel file

Why batch association resolution: a naive approach would call the API once per deal to resolve company/contact names: that's N API calls for N deals. Instead, all unique IDs across all deals are collected first, then resolved in batches of 100. For 1000 deals with associations, that's ~10 calls instead of 1000. Same data, 100× fewer requests

04 - Design

Why it was built this way

Key technical choices and the reasoning behind them

Pagination
Cursor-based deal retrieval

Chosen

While-loop with cursor extracted from paging.next.after, stops when key absent

Why

HubSpot caps results at 100 per request. Cursor pagination is the only correct approach: offset pagination would miss records inserted between pages

Property Discovery
Dynamic property resolution vs hardcoded list

Chosen

Query /crm/v3/properties/deals at startup, filter to schema-required columns

Why

HubSpot property names can change. Dynamic resolution decouples the extraction from the API's internal naming: the reference file is the source of truth, not a hardcoded list

Type Coercion
Reference-driven type enforcement

Chosen

Loop through ExampleDf column dtypes, apply matching conversion to DealsDf

Why

Explicit per-column type logic in a hardcoded switch would need updating every time a column is added. Deriving from the reference file makes the type enforcement self-maintaining

Datetime
String serialization of datetime output

Chosen

dt.strftime("%Y-%m-%d %H:%M:%S") after timezone conversion, before Excel export

Why

Excel datetime handling with timezone-aware columns produces inconsistent output across pandas versions and Excel clients. String serialization is deterministic and portable
