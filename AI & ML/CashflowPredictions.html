01 - Problem

Revenue forecasting across heterogeneous business units

A single forecasting model cannot serve 20+ teams with fundamentally different revenue patterns, seasonality, and macro sensitivity. The challenge is building a system that adapts

Naive approaches

  • |Single aggregate forecast hides team-level variance: CFO can't act on it
  • |Prophet alone ignores macroeconomic context: interest rate shocks and inflation cycles aren't in the time series
  • |Linear regression alone can't model the non-linear seasonality patterns in monthly revenue
  • |Outlier months (COVID, exceptional contracts) distort model fits without robust preprocessing
  • |No per-team diagnostic: impossible to audit which teams drive forecast error

This system

  • →Independent model fit per team: each team gets its own Prophet + Regression pipeline
  • →6 macroeconomic regressors enrich the regression layer with external economic signal
  • →Prophet handles seasonality (yearly, monthly, quarterly, weekly, daily) + Italian holiday calendar
  • →Winsorization clips outliers before fitting
  • →Interactive ipywidgets dashboard: team-by-team AE, APE, and cumulative MAPE diagnostics

Conditional ensemble logic: blending Prophet and Regression only when the regression MAPE is within ±25% is the key design decision. A poorly-fitting regression makes the ensemble worse than Prophet alone. The gate ensures that for teams where macroeconomic regressors have low predictive power, Prophet forecasts unaided and the regression benefit is only claimed where it's empirically justified.

02 - Architecture

Prophet + Ensemble Regression

Two model families run in sequence. Prophet forecasts from the time series. The regression layer adds macro signal. The conditional gate decides whether they blend

Regression Ensemble

🎯
Stage 3
Conditional Ensemble
Average(Prophet, Regression) if MAPE ≤ 25%, else Prophet only

Conditional ensemble gate

Gate passes — MAPE ≤ 25%

Regression has empirical predictive power for this team. Simple average of Prophet and Regression forecasts. Error metrics (AE, APE) are also averaged in the same way

Gate fails — MAPE > 25%

Macroeconomic regressors don't add signal for this team. Prophet forecasts alone. Blending a poor regression model would degrade accuracy rather than improve it

03 - Pipeline

Data to forecast in four stages

Raw revenue data and macroeconomic inputs are cleaned, modelled, combined, and visualised through an interactive diagnostic dashboard

Data preparation per team
  • Revenue loaded from Oracle DB
  • TeamMap normalizes 40+ raw team names into 20+ clean groups
  • Merge with macroeconomic DataFrame on date
  • Winsorize each numeric column
  • Zero-revenue teams excluded before modelling
02
Prophet Forecast
Time-series decomposition
  • Logistic growth with cap/floor per team
  • Multiplicative seasonality mode
  • 5 seasonality components: yearly, monthly, quarterly, weekly, daily
  • Italian public holiday calendar injected
  • 24-month future dataframe; confidence interval 70%
03
Regression Ensemble
Macro-aware layer
  • Features: macro regressors + year + month
  • StandardScaler → PCA (5 components) → PolynomialFeatures (deg 2)

  • MAE-inverse weighting: better model gets higher share
  • Ensemble predictions for test + new (forecast) rows

Diagnostics + dashboard
  • Prophet and Regression outputs merged on ds_team key
  • Conditional gate applied per team row
  • 3-panel plot: revenue forecast + AE + APE over time

04 - Regressors

Six external signals, one regression layer

Each regressor was selected for its potential causal relationship with corporate revenue in the Italian market. All are quarterly figures interpolated to monthly frequency

Forward imputation for the forecast horizon: macroeconomic regressors are only available up to the last historical quarter. For the 24-month forecast horizon, each column is forward-filled using a 24-month rolling mean per team, a conservative extrapolation that assumes recent macro conditions persist, rather than requiring external economic projections as input.

05 - Decisions

Design choices that matter

Key tradeoffs in preprocessing, model selection, and diagnostic design

Preprocessing

Winsorization over Removal

Outlier months (COVID shutdowns, exceptional contracts) are winsorized at 5%–7.5% rather than dropped. Dropping creates gaps that break Prophet's time-series continuity. Winsorizing clips the extremes while preserving the time structure

scipy.stats.mstats.winsorize applied per column per team
Lower limit 5%, upper limit 7.5%: asymmetric to preserve growth signal
Applied after zero-team removal to avoid clipping on sparse series
Prophet Config

Logistic Growth + 5 Seasonalities

Logistic growth with a per-team cap and floor prevents unbounded forecasts — revenue can't grow indefinitely. Five seasonality layers (yearly, monthly, quarterly, weekly, daily) capture the full cyclical structure of corporate billing

changepoint_range=1: changepoints anywhere in history
Interval width 70% for confidence bands
Italian holidays calendar from the holidays library
Regression Design

PCA + Polynomial + MAE-Weighting

12 raw features compressed to 5 PCA components before polynomial expansion — controls dimensionality explosion from degree-2 terms. Ridge and Lasso are each fit with CV; their blend weight is the inverse of their test MAE

PCA n_components=5: retains most variance, removes collinear macro signals
PolynomialFeatures degree=2: captures non-linear macro interactions
MAE-inverse weighting: better-fit model automatically gets more weight
