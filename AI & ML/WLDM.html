01 - Problem

Crawl everything

Finding topic-relevant pages across hundreds of competitor and partner domains is a needle-in-a-haystack problem. Naive approaches are too slow, too expensive, or too imprecise to scale to an analyst workflow

Naive approaches

  • |Full crawl is slow, expensive, and gets you rate-limited or blocked
  • |Keyword matching on slugs alone misses multilingual content and product names that never appear in URLs
  • |No deduplication: regional language variants flood the candidate list with near-identical pages
  • |SPA-rendered sites (React, Next.js) return empty HTML shells to non-JS clients
  • |Per-domain tuning doesn't scale: analysts spend hours per site, not minutes

This pipeline

  • →Sitemap-first discovery: most sites hand you their URL inventory for free: crawl is the fallback, not the default
  • →Sentence embeddings catch semantic neighbors that lexical filters miss, across languages and brand names
  • →rapidfuzz dedup drops near-identical URLs before scoring touches them
  • →Playwright SPA fallback renders JS-heavy homepages and harvests hydrated links
  • →Stage-isolated Parquet caches: tune the scorer without re-crawling; failed runs resume mid-pipeline

Core design insight: crawling is expensive because most crawlers don't ask the site first. Sitemaps and robots.txt are free inventories that cover the long tail of URLs. Treating crawl as the fallback slashes network cost and latency before a single embedding is computed.

02 - Architecture

Four stages, each independently re-runnable

Every stage writes its own Parquet to cache. Resume failed runs without replaying completed stages. Each output is a clean contract for the next

🗺️
Stage 1
Discover
sitemap + robots.txt + BFS crawl fallback
🎭
Stage 1b
SPA Fallback
Playwright renders JS-heavy shells
🔍
Stage 2
Filter
regex boilerplate drop + rapidfuzz dedup
🧠
Stage 3
Score
embed · cosine sim · lexical blend · gate

Why Parquet for intermediate state: columnar, compressed, typed: and crucially, stage outputs survive process restarts. Full runs against hundreds of domains can take tens of minutes; a mid-run failure shouldn't mean restarting from zero. Every stage reads the previous stage's Parquet and writes its own. Tuning the scorer doesn't re-crawl the web.

03 - Pipeline

Domains in, ranked URLs out

Each stage has a clear responsibility and a clean handoff. Smoke mode (first 20 domains) lets you tune thresholds cheaply before paying full crawl cost

01
Discover
Sitemap-first URL inventory
  • Resolves http/https × bare/www variants
  • Parses robots.txt for Sitemap
  • Inspects <link rel="sitemap"> on homepage
  • Probes 20+ common sitemap paths
  • Recursively expands sitemap indexes
  • Falls back to depth-1 BFS crawl
01b
SPA Fallback
Playwright headless pass
  • React / Next.js sites ship empty shells
  • Playwright renders homepage fully
  • Harvests links from hydrated DOM
  • Separate stage: ~10× slower, used sparingly
  • Only triggers for domains with no URLs after Stage 1
02
Filter
Boilerplate drop + dedup
  • Drops /privacy, /terms, /login, /cart, /wp-admin, asset extensions
  • rapidfuzz ratio ≥ 85 → language/region duplicates removed
  • Caps at MaxCandidatesPerDomain=50
  • Strong-token override bypasses boilerplate filter for high-signal slugs
03
Score
Embed · cosine · blend · gate
  • Async-fetches each candidate page
  • selectolax extracts title, H1, meta description
  • Keywords embedded once, cached to KeywordEmbeddings.npy
  • Page text embedded in batches of 64
  • Cosine sim → max across keyword matrix
  • Final: 0.7 × semantic + 0.3 × lexical

Strong-token override: URLs containing high-signal gambling slugs (casino, bet, bonus, megaways, ...) bypass the boilerplate drop regardless of path pattern. This prevents precision loss on domains where the product vocabulary overlaps with generic path structure: a real problem in the gambling vertical where /bonus might otherwise be filtered as a generic marketing page.

04 - Scoring Model

Semantic + lexical

Neither pure embedding similarity nor pure keyword matching wins alone. The blend outperforms either in isolation on this dataset. Two gates keep precision high at the tail

LEXICAL 30%

Semantic layer

all-MiniLM-L6-v2

384-dimensional sentence embeddings via sentence-transformers. Keywords embedded once and cached to disk: zero re-embedding cost across runs

Strong quality-per-MB: runs CPU-only at thousands of docs/min
Batch size 64 for page text
Cosine sim against full keyword matrix → takes the max
Catches multilingual content and brand names missing from slugs
Lexical layer

Token presence bonus

A literal keyword token appearing in the page title, H1, or URL slug adds a normalized bonus to the final score. Cheap signal, high precision when present

No regex overhead
Applies to title, H1, and slug independently
Normalized before blending: doesn't overwhelm semantic score
Especially effective on exact-match gambling product names
Precision gates

Floor + gambling gate

Two hard filters prevent low-quality rows from polluting the output, keeping analyst review time low even at scale

SemanticFloor=0.35
Title/H1 gambling gate: rows scoring < 0.80 must contain a strong gambling token in title or H1
Strong-token list is auditable and hand-tunable in Config.py
Blend weight (0.7/0.3) and both thresholds are smoke-mode tunable
05 - Design

Every choice has a reason

Key tradeoffs made explicit

URL discovery
Sitemap-first, crawl-fallback
vs crawl-first

Chosen

Parse robots.txt, probe 20+ sitemap paths, expand sitemap indexes: only fall back to BFS crawl if no sitemap found

Why

Crawling is expensive because it's unnecessary most of the time. Sites hand you their URL inventory for free if you ask. Crawl-first means paying full network cost for information that's already published and risking rate-limit bans

HTML parsing
selectolax over lxml / BeautifulSoup
vs standard parsers

Chosen

selectolax for title, H1, and meta description extraction across every candidate page

Why

~5× faster than BeautifulSoup on title/H1/meta extraction. At 50 URLs × hundreds of domains, the difference is minutes of wall-clock time per run

Deduplication
rapidfuzz ratio ≥ 85
vs exact match or string edit distance

Chosen

rapidfuzz fuzzy ratio to detect and drop near-duplicate URLs (language/region variants like /en/ vs /it/)

Why

Exact match misses the /en-gb/ vs /en-us/ problem. Pure edit distance is slow in Python at scale. rapidfuzz's C++ backend runs orders of magnitude faster than fuzzywuzzy with the same interface and ratio=85 is well-calibrated for URL variants without over-collapsing

Async I/O
httpx + asyncio, single process
vs threads or multiprocessing

Chosen

httpx async with GlobalConcurrency=40 and PerHostRatePerSec=1.5 for all network I/O

Why

URL discovery and page fetching are I/O-bound, not CPU-bound. Single-process async avoids multiprocessing overhead and shared-state complexity. Per-host rate limiting is a first-class concern: httpx limits are per-host, not global, which keeps individual domains from getting blocked.

Embedding model
all-MiniLM-L6-v2 (384-d)
vs larger models or API embeddings

Chosen

sentence-transformers/all-MiniLM-L6-v2, run locally, keyword embeddings cached to .npy

Why

Strong quality-per-MB ratio, runs CPU-only at thousands of docs per minute. API embeddings add latency, cost, and a network dependency per run. Caching keyword embeddings to disk means zero re-embedding cost when only the domain list or thresholds change.
